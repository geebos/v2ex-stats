## 任务三：利用 Chrome Sync Storage 实现跨设备数据同步

### 1. 需求概述

当前所有统计数据（在线时长、帖子浏览记录等）均存储在 `local storage` 中，仅对当前设备可用。对于已登录 Chrome 账号的用户，可通过 `chrome.storage.sync` 将核心数据同步到 Google 账号，实现跨设备数据共享。由于 sync storage 有严格的容量限制（总量 100KB、单条 8KB、每分钟写入不超过 1800 次），需要只同步精简后的核心数据，历史全量数据仍保留在 local storage。

### 2. 同步数据范围

同步 payload 与现有 local storage 中的**在线时长、帖子状态数据结构一致**，仅**存储位置**改为 `chrome.storage.sync`；每条同步数据需增加 **版本号字段**（如 `syncVersion`），用于冲突检测与后续 schema 迁移。

| 数据类型 | 字段说明 | 存储策略 |
|----------|----------|----------|
| 在线时长统计 | 以 **30 秒**为统计单位；本地维护**缓存队列**，每触发一个单位向队列追加一个**时间戳**（当前分钟 0–30 秒取整到 :00，30–60 秒取整到 :30）；合并时对时间戳去重后计数即得多设备在线时长。**小时粒度**的汇总数据增加时间戳作为版本号，用于过滤过时数据 | 去重后的时间戳集合序列化写入 `sync:onlineTime`；小时粒度数据带时间戳版本号 |
| 最近三天帖子数据 | 与现有一致：近 3 天浏览/忽略的帖子 id 及标题摘要；payload 含 `syncVersion` | 按日期分 key 写入，每天一条 `sync:posts:YYYYMMDD` |
| 忽略帖子列表 | 与现有一致：用户主动忽略的帖子 id 集合；payload 含 `syncVersion` | 写入 `sync:ignoredPosts`，超出限制时截断最旧的记录 |

**在线时长合并算法**：

- **统计单位**：以 30 秒为一个单位进行累计。
- **缓存队列**：每次触发一个单位（满 30 秒在线）时，向本地缓存队列追加一个**时间戳**；时间戳取值规则：当前分钟 0–30 秒选该分钟整点（`:00`），30–60 秒选该分钟半（`:30`），即同一 30 秒区间在多设备上产生相同时间戳。
- **合并**：从 sync 与本地拉取时间戳集合后**去重**，去重后的数量 × 30 秒即为多设备合并后的在线时长；合并结果写回 local 与 sync storage。
- **小时粒度与版本号**：若存在按小时汇总的数据，为每条小时粒度数据增加**时间戳字段**作为版本号，同步与合并时根据该时间戳过滤过时数据后再去重、计数并更新到 sync storage。

| 功能点序号 | 功能点名称 | 功能点描述 | 前置依赖 |
|------------|------------|------------|----------|
